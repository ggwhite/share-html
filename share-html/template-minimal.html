1

背景與目標

說明為何發起本次選型，以及需要滿足的核心需求。

核心需求

高並發 API 服務

目標系統需支援每秒萬級請求，P99 延遲控制在 50ms 以內。同時需要完整的可觀測性支援（tracing、metrics、logging），以及與現有 Go 微服務生態的互通性。

2

方案比較

針對三個候選方案進行多維度量化對比。

功能 / 指標 方案 A — Gin 方案 B — Fiber ✦
RPS（單核） ~85,000 ~140,000
記憶體佔用 中 低（零分配路由）
生態成熟度 高，middleware 豐富 中，快速成長中
net/http 相容 完全相容 部分相容（fasthttp）
INFO

Fiber v3 已進入 RC 階段，net/http 相容層在新版本中有顯著改善，可於下季重新評估。

WARN

fasthttp 的 RequestCtx 並非 context.Context，既有 middleware 若直接移植需逐一驗證相容性。

OK

建議採用 Gin 作為本期主力框架，等 Fiber v3 GA 後於新服務試點，成熟後再逐步遷移。

DANGER

請勿在本季將任何已上線服務切換至 Fiber，fasthttp 的 goroutine 洩漏問題在高負載下尚未完全排除。
